# 下一阶段计划：手机扫码、型号化编码与打标

## 已确认决策

- 编码格式选择：`型号-递增编号`
- 示例：
  - `GM6020-0001`
  - `GM6020-0002`
  - `M3508-0001`
- 每种型号单独递增编号。
- 已经使用过的编号不复用，即使电机报废或作废，也继续往后编号。
- 扫码方式优先改为手机网页扫码 H5/PWA。
- 暂时不优先做微信小程序，等网页扫码流程稳定后再考虑。
- 打标机可以作为标签输出设备。
- 当前打标机不能联网，只能本地使用。
- 当前打标机支持 Code128 条形码。
- 当前打标机使用方式是：本地软件输入文字，然后打印条形码。
- Code128 打出来可能偏长，当前阶段可以先接受；后续可切换为二维码以减小标签尺寸。

## 编码规则计划

### 目标格式

`MODEL-0001`

其中：

- `MODEL` 来自电机型号，例如 `GM6020`
- `0001` 是该型号下的递增流水号

### 例子

| 型号 | 第 1 台 | 第 2 台 | 第 12 台 |
|---|---|---|---|
| GM6020 | GM6020-0001 | GM6020-0002 | GM6020-0012 |
| M3508 | M3508-0001 | M3508-0002 | M3508-0012 |
| M2006 | M2006-0001 | M2006-0002 | M2006-0012 |

## 手机扫码计划

### 推荐方式

先做手机网页扫码页面，而不是先做小程序。

原因：

- 可以复用现有 Next.js 系统
- 手机和电脑访问同一个内网服务
- 开发和调试速度更快
- 不需要先处理微信小程序发布、授权、审核等问题

### 页面计划

- [ ] 手机扫码入口页
- [ ] 手机扫码入库页
- [ ] 手机扫码出库页
- [ ] 手机扫码后识别电机详情
- [ ] 扫码失败或编码不存在时给出明确提示
- [ ] 入库成功后显示成功结果
- [ ] 出库成功后显示成功结果

## 打标计划

### 打标机是否可以用

可以用，而且长期来看更推荐使用打标机。

原因：

- 标签更耐磨、耐油污、耐撕扯
- 更适合贴在电机或收纳盒上
- 标签尺寸更稳定
- 批量打印效率更高

当前限制：

- 现有打标机不能联网，系统不能直接通过网络控制它。
- 因此第一阶段不做“系统一键直连打标机打印”。
- 系统应先生成可复制、可导出、可被本地打标软件使用的编码和条形码素材。

### 系统侧优先实现

第一步先做系统内的“标签打印页”：

- [ ] 电机详情页增加“打印标签”按钮
- [ ] 标签页展示电机型号、编码、二维码
- [ ] 标签页适配小尺寸打印布局
- [ ] 支持浏览器打印

### 打标机适配方式

根据打标机能力选择：

| 方式 | 说明 | 推荐阶段 |
|---|---|---|
| 浏览器打印 | 打标机被电脑识别成普通打印机，网页直接打印 | 第一优先 |
| PDF 打印 | 系统生成标签 PDF，再用打标机软件打印 | 第二优先 |
| 图片导出 | 系统导出二维码/标签图片，再导入打标机软件 | 备用 |
| CSV/Excel 导出 | 系统导出编码列表，本地打标软件批量导入打印 | 当前打标机优先评估 |
| SDK/API 对接 | 直接调用打标机接口打印 | 后期再做 |

现有打标机只能通过本地软件使用，并支持“输入文本后打印 Code128 条形码”。因此当前优先流程为：

1. 系统生成 `GM6020-0001` 这类编码。
2. 系统在详情页提供“复制编码”按钮。
3. 用户把编码粘贴到打标机本地软件。
4. 打标机软件按 Code128 打印条形码标签。
5. 后续如果 Code128 标签过长，再增加二维码标签模式。

## 最少外设原则

目标是尽量少依赖额外设备：

### 最小设备组合

- 电脑或内网服务器：运行系统
- 手机：扫码入库、扫码出库
- 打标机或普通打印机：只在首次建档贴标签时使用

### 实际流程

1. 电脑端新建电机档案
2. 系统生成 `GM6020-0001`
3. 系统生成二维码标签
4. 用打标机打印标签
5. 把标签贴到电机或收纳位置
6. 后续用手机扫码完成入库、出库、查询
