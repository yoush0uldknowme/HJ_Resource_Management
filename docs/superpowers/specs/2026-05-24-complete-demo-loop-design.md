# 电机资源管理完整 demo 闭环版设计

## 目标

把当前电机资源管理 MVP 推进成一版可以完整演示预期工作方式的 demo。这个版本不再只做部分流程测试，而是先让电脑端建档、标签打印、手机端扫码查询、扫码入库、扫码出库和电脑端追踪记录形成完整闭环，再拿去现场试跑并收集真实反馈。

## 背景

当前系统已经具备登录、首页统计、电机建档、按型号生成内部编码、详情页二维码和 Code128 预览、复制编码、列表查询、入库、出库、日志记录等基础能力。现阶段主要问题不是技术骨架缺失，而是产品闭环还不完整：

- 页面和文档存在大量中文乱码，现场人员无法正常理解。
- 标签打印还停留在详情页预览，没有独立的打印页面。
- 入库和出库页面更像后台表单，扫码成功、失败、重复操作等反馈不够清楚。
- 手机端扫码查询、入库、出库入口还没有落地，无法演示最终设想的现场工作方式。

## 范围

### 本版本要做

- 修复主要页面、组件、业务提示、README 和进度文档中的中文乱码。
- 保持现有电脑端管理入口：登录、首页、电机列表、新建电机、详情、入库、出库、日志。
- 增加电脑端独立标签打印页，从电机详情页进入。
- 标签打印页展示电机型号、内部编码、二维码和 Code128，并提供适合浏览器打印的样式。
- 改进入库和出库操作反馈：成功后显示明确结果，失败时显示可理解的原因。
- 增加手机端入口页，面向现场扫码操作。
- 增加手机端扫码查询页面，扫到或输入编码后展示电机摘要。
- 增加手机端扫码入库页面，扫到或输入编码后完成入库并显示结果。
- 增加手机端扫码出库页面，扫到或输入编码后填写出库人、使用车辆并完成出库。
- 手机端在摄像头扫码不可用时，必须保留手动输入编码的降级方式。
- 保留现有 SQLite、Prisma、Next.js App Router 和本地图片存储方案。

### 本版本不做

- 不做微信小程序。
- 不做打标机直连。
- 不做完整权限体系。
- 不做电机编辑、删除、作废流程。
- 不做数据导出和自动备份。
- 不扩展到裁判系统、打印耗材或其他物资模块。
- 不做复杂审批流程。

## 核心演示流程

1. 管理员在电脑端登录系统。
2. 在电脑端新建电机档案，填写名称、型号和备注，可上传首张照片。
3. 系统按型号生成唯一编码，例如 `GM6020-0001`。
4. 电脑端进入详情页，查看二维码和 Code128，复制编码。
5. 电脑端进入标签打印页，打印或把编码复制到本地打标软件生成标签。
6. 标签贴到电机或收纳位置上。
7. 手机端打开现场入口，扫码或输入编码查询电机。
8. 手机端执行扫码入库，成功后显示电机编码、状态和下一步入口。
9. 手机端再次扫码执行出库，填写出库人和使用车辆。
10. 电脑端列表、详情页和日志页能看到状态变化和完整流转记录。

## 页面结构

### 电脑端

- `/login`：登录页。
- `/`：首页概览，展示总数、在库、已领用、待入库和最近更新。
- `/motors`：电机列表，支持编码、型号、SN、名称和状态筛选。
- `/motors/new`：新建电机档案。
- `/motors/[id]`：电机详情，展示基础信息、照片、二维码、Code128 和流转记录。
- `/motors/[id]/label`：标签打印页。
- `/motors/inbound`：电脑端入库页。
- `/motors/outbound`：电脑端出库页。
- `/logs`：全局操作记录页。

### 手机端

- `/mobile`：手机现场入口。
- `/mobile/scan`：扫码查询入口，支持摄像头扫码和手动输入。
- `/mobile/inbound`：手机入库。
- `/mobile/outbound`：手机出库。

手机端页面应使用更大的按钮、输入框和结果状态，减少表格，优先让现场人员在手机屏幕上一眼看清操作结果。

## 数据和业务规则

- 电机编码继续使用 `型号-四位递增序号`，例如 `GM6020-0001`。
- 每种型号独立递增。
- 已使用过的编码不复用。
- 建档后默认状态为 `draft`，界面显示为 `待入库`。
- 入库后状态为 `in_stock`，界面显示为 `在库`。
- 出库只允许从 `in_stock` 状态执行，成功后状态为 `checked_out`，界面显示为 `已领用`。
- 入库流水记录 `transactionType = inbound`。
- 出库流水记录 `transactionType = outbound`，记录出库人和使用车辆。
- 查询可以通过 `motorCode` 或 `snCode` 匹配。
- 二维码和 Code128 的内容都使用内部编码。

## 错误和反馈

电脑端和手机端都应避免直接抛出技术错误页面。需要覆盖以下场景：

- 编码为空：提示需要扫码或输入编码。
- 编码不存在：提示未找到对应电机。
- 已出库电机再次出库：提示只有在库电机可以出库，并显示当前状态。
- 入库成功：显示编码、状态和查看详情入口。
- 出库成功：显示编码、出库人、车辆和查看详情入口。
- 摄像头扫码不可用：提示可以手动输入编码继续操作。

## 技术方案

- 继续使用 Next.js App Router。
- 继续使用 Server Actions 处理表单提交。
- 复用 `applyInbound` 和 `applyOutbound` 作为业务状态转换核心。
- 抽出可复用的扫码/编码查询辅助函数，供电脑端和手机端共享。
- 标签页复用详情页已有的 QRCode 和 Code128 生成逻辑。
- 使用 CSS `@media print` 为标签打印页提供打印布局。
- 摄像头扫码优先使用浏览器能力；如果实现风险过高，先完成手动输入降级和可替换的扫码组件边界。

## 验收标准

- `npm run lint` 通过。
- `npm test` 通过。
- `npm run build` 通过。
- 主要页面中文可读，没有明显乱码。
- 电脑端能完成建档、查看详情、打开标签页、入库、出库、查看日志。
- 标签打印页能展示编码、二维码和 Code128，并能触发浏览器打印。
- 手机端能打开入口页。
- 手机端能通过手动输入编码完成查询、入库和出库。
- 手机端扫码能力如果可用，应能读取标签编码；如果不可用，页面仍能通过手动输入完成完整 demo。
- 失败场景显示用户可理解的提示，而不是技术错误页。

## 后续测试策略

完整 demo 闭环完成后，再拿真实电机和真实标签做现场测试。现场重点观察：

- 标签尺寸是否合适。
- 手机扫码成功率是否足够。
- 入库和出库步骤是否顺手。
- 出库人和车辆字段是否足够。
- 列表照片是否能帮助识别电机。
- 现场人员是否需要编辑、作废、导出或备份能力。
